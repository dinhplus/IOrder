# ADR 0005: QR Code Table Ordering

## Status

Accepted

## Date

2026-03-02

## Context

QR code table ordering is a core feature of IOrder. The QR flow must be designed to ensure:

1. **No app installation required** — customers only need a phone camera
2. **Security** — QR codes cannot be forged or shared between tables
3. **Smooth UX** — time from scan to menu display < 2 seconds
4. **Multi-device session** — multiple people at the same table can view and order simultaneously
5. **QR expiry management** — expired QR codes can be regenerated by staff

## Decision

### QR URL Structure

```
https://order.iorder.vn/{restaurant_slug}/table/{table_id}?token={jwt}
```

Example:
```
https://order.iorder.vn/pho-ha-noi/table/tbl_abc123?token=eyJhbGciOiJIUzI1...
```

### JWT Token Design

The QR token is a **JWT HS256** signed with `TABLE_QR_SECRET` (a per-restaurant secret stored in AWS Secrets Manager):

```json
// Header
{
  "alg": "HS256",
  "typ": "JWT"
}

// Payload
{
  "sub": "table:{table_id}",
  "tid": "{tenant_id}",           // tenant ID
  "tbl": "{table_id}",            // table ID
  "iat": 1709340000,              // issued at
  "exp": 1709361600,              // expires in 6 hours
  "jti": "{uuid}"                 // unique ID (prevents replay attacks)
}
```

**Token lifetime: 6 hours** — sufficient for the longest dinner service.

### Token Validation Flow

```
Customer scans QR → Browser opens URL
         │
         ▼
Frontend sends GET /api/v1/qr/validate?token={jwt}
         │
         ▼
Backend validates:
  1. JWT signature (HMAC-SHA256 with restaurant secret)
  2. Token not expired (exp > now)
  3. Table exists and is active
  4. Restaurant is currently open (operating hours)
         │
    ┌────┴────┐
   VALID    INVALID
     │           │
     ▼           ▼
Return      Return 401:
session     "QR code expired.
info        Please ask a staff
             member to refresh."
     │
     ▼
Create/Resume table session
Return: { sessionId, tableId, tenantId, menuUrl }
```

### Session Management

A **table session** represents one service period at a table (from first scan to bill closure):

```go
type TableSession struct {
    ID        string    // session UUID
    TableID   string
    TenantID  string
    Status    string    // active, closed
    OpenedAt  time.Time
    ClosedAt  *time.Time
}
```

- Valid QR → find or create an active session for that table
- Multiple devices scanning the same QR → same session → real-time sync of the cart
- Staff resets table → old session closed; new session created when QR is scanned next

### Multi-Device Ordering

Multiple people at the same table can order simultaneously:
- All devices join the same `sessionId` WebSocket channel
- Cart updates are synced in real time across all devices
- One person adding an item → all others see the update immediately
- Configurable cart mode: **Shared cart** (single combined cart) or **Individual cart** (separate cart per device, enabling split billing)

### QR Regeneration

Staff can regenerate the QR when:
- The existing QR has expired (> 6 hours)
- The table is reset after a guest departs
- The QR has been distributed outside the restaurant (security concern)

```
POST /api/v1/tables/{tableId}/qr
→ Generate new JWT with new jti
→ Update table.qr_token in the database
→ Return new QR image (base64 PNG) and download URL
→ Old token is invalidated (jti blocklist in DynamoDB)
```

### QR Generation & Rendering

- QR is rendered server-side using the Go library `github.com/skip2/go-qrcode`
- Output formats: PNG (300×300), PDF A4 (6 QR codes per page with restaurant branding)
- Error correction level: **M** (15% data recovery) — adequate for laminated sticker QR codes
- QR image is stored in S3 with a signed URL that has an expiry

### Security Considerations

1. **Non-reusable after reset**: When staff resets a table, a new `jti` is generated → old token becomes invalid
2. **Rate limiting**: The validate endpoint is limited to 10 requests/minute per IP
3. **HTTPS only**: QR URLs must use HTTPS; HTTP redirects are not permitted
4. **Unpredictable table IDs**: `table_id` is a UUID, not a sequential integer
5. **Tenant isolation**: Token contains `tenant_id`; server verifies the token belongs to the correct tenant

### PWA vs. Native App

A **Progressive Web App (PWA)** was chosen for the customer ordering flow because:
- No app installation → minimum friction for customers
- Camera scan → opens directly in the browser
- PWA can be "Added to Home Screen" to support member retention
- Instant updates (no App Store review required)

The native React Native app is still developed for:
- Staff/KDS interface (reliable push notifications)
- Restaurant owner dashboard on mobile
- Advanced features: offline mode, biometric authentication

## Consequences

### Positive

- Zero friction for customers: only a camera is needed
- Secure: JWT is signed, has expiry, and uses a per-restaurant secret
- Multi-device ordering within the same session
- Staff have clear controls (reset, regenerate)
- PWA loads fast and requires no installation

### Negative

- 6-hour token lifetime: server clock drift could invalidate tokens (requires NTP sync)
- Internet-dependent: customers need a connection to view the menu (offline mode planned for v2.0)
- QR stickers may become dirty or damaged — laminated backup QR codes are recommended

### Mitigation

- NTP sync is automatic on ECS Fargate (AWS managed)
- CDN caching of menu data reduces latency
- Provide a backup PDF of QR codes for restaurants to print additional copies
